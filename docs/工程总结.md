### Version 2.40 - Chinese/CJK Input Fix (2026-02-28)

**Bug Fix ‚Äî Terminal Chinese/CJK IME Input Garbled:**

Root cause: `attachCustomKeyEventHandler` in `Terminal.tsx` was processing keydown events during IME composition without checking `event.isComposing` or `event.keyCode === 229`. This interfered with xterm.js internal `CompositionHelper`, causing Chinese input to be garbled in both display (echo) and execution.

Fix: Added IME composition guard at the top of the custom key handler:
```typescript
if (event.isComposing || event.keyCode === 229) {
  return true  // Let xterm.js CompositionHelper handle IME events
}
```

**Bug Fix ‚Äî Windows Terminal UTF-8 Output Truncation:**

`pty_windows.go` was missing `UTF8SafeBuffer` for output reading. When `Read()` splits a multi-byte UTF-8 character at buffer boundary, `string(buffer[:n])` creates invalid UTF-8, causing garbled output. Fixed by adding `UTF8SafeBuffer` (matching `pty_unix.go` pattern) with `AppendAndFlush()` and deferred `Flush()`.

**Enhancement ‚Äî SSH Terminal UTF-8 Locale:**

Added `Setenv("LANG", "en_US.UTF-8")` and `Setenv("LC_ALL", "en_US.UTF-8")` after SSH PTY request. Most SSH servers reject `Setenv` for security, so errors are silently ignored. This helps when the remote server has a non-UTF-8 default locale.

**Files Changed:**
- **Frontend**: `frontend/src/components/terminal/Terminal.tsx` ‚Äî IME composition guard (+8 lines)
- **Backend**: `internal/app/pty_windows.go` ‚Äî UTF8SafeBuffer for Windows ConPTY output (+13 lines)
- **Backend**: `internal/app/terminal_handler.go` ‚Äî SSH session UTF-8 locale env vars (+7 lines)

**Key Lessons:**
- xterm.js `attachCustomKeyEventHandler` runs BEFORE internal `CompositionHelper.keydown()` ‚Äî must explicitly skip IME events
- `event.isComposing` and `event.keyCode === 229` are the standard ways to detect IME composition in progress
- All terminal output readers (SSH, Unix local, Windows local) must use `UTF8SafeBuffer` to handle multi-byte character splitting at buffer boundaries

`CopyRemoteFilesToSystemClipboard` previously called only `DownloadFile()` which fails on directories (SFTP `Open()` rejects dirs). Now uses `sftpClient.Stat()` to detect file vs directory and routes to `DownloadFile` or `DownloadDirectory` accordingly.

**Critical Bug Fix ‚Äî Batch Close Only Removing One Tab:**

When using "Close All to the Left/Right/Others", only one tab was actually removed from the UI. Backend resources (SSH/PTY) were cleaned up correctly (tabs turned red = disconnected), but the tab elements remained.

Root cause: Each `handleCloseSession()` call reads `sessions` from a React closure snapshot. In a `forEach` loop, every call operates on the same stale array. `setSessions(remainingSessions)` is async ‚Äî subsequent iterations still use the original array, so only the last `setSessions` call takes effect.

Solution: Replaced `forEach(handleClose)` loops with dedicated `batchClose*()` functions that compute the final remaining array first, clean up backend resources for all removed items, then call `setSessions(remaining)` exactly once.

- **TerminalTab**: `batchCloseSessions(keepFilter)` ‚Äî cleans up PTY/SSH connections, one `setSessions()` + `persistSessions()`
- **EditorTab**: `batchCloseFiles(keepFilter)` ‚Äî unmodified files removed in one `setFiles()`, modified files individually prompted via `Modal.confirm`

### Version 2.37 - Drag & Drop Fix & Debug Log Tab (2026-02-09)

**Critical Bug Fix ‚Äî Wails WKWebView Drop Event Never Fires:**

This was a complex debugging session that uncovered a fundamental limitation of Wails on macOS.

When `DragAndDrop.DisableWebViewDrop: true` is set in `main.go`, the native WKWebView (`WailsWebView.m:performDragOperation`) intercepts ALL drop operations at the Objective-C level. This means the JavaScript `drop` event **never fires** for ANY drag operation ‚Äî not for OS-level Finder drags, and not for in-app HTML5 drags between components. Only `dragend` fires.

**Symptoms observed:**
1. Dragging files from file manager to terminal: overlay appeared ("ÊãñÂä®Êñá‰ª∂‰ª•ÊèíÂÖ•Ë∑ØÂæÑÂà∞ÁªàÁ´Ø") but releasing mouse did nothing
2. Dragging files between file managers (remote‚Üîlocal): no upload/download occurred
3. File manager "drag-over" highlight CSS stayed permanently after releasing mouse

**Root cause analysis path:**
- Initially suspected `dataTransfer.getData()` returning empty (WKWebView restriction) ‚Üí partially true but not the core issue
- Added shared memory module (`dragState.ts`) to bypass `dataTransfer` ‚Üí still didn't work
- Added in-app **Log Tab** for debugging (since Wails production builds have no DevTools) ‚Üí revealed that `drop` event NEVER fires, only `dragend` fires
- Read Wails source code (`WailsWebView.m`) and confirmed: `disableWebViewDragAndDrop=true` causes native layer to handle drop, JS `drop` event is suppressed

**Solution ‚Äî `dragend` + shared memory + `elementFromPoint`:**
1. **`dragState.ts`**: Shared module stores `DragPayload` (source + path) and `DragTarget` (terminal / local-fm / remote-fm)
2. **`dragStart`**: File managers call `setDragPayload()` to store path in memory
3. **`dragover`** (capture phase on Content): Uses `document.elementFromPoint()` to detect which zone cursor is over, calls `setDragTarget()`
4. **`dragend`** (window capture phase): Reads payload + target from shared memory, dispatches action:
   - `terminal` ‚Üí `WriteToTerminal()` with escaped path
   - `local-fm` + remote source ‚Üí `DownloadFile()` (SFTP download)
   - `remote-fm` + local source ‚Üí `UploadFile()` (SFTP upload)
5. **Visual state cleanup**: File managers listen for `dragend` on `window` to clear `dragOver` CSS state, since `drop` (which normally clears it) never fires

**Other fixes:**
- `ConnectSSH()` was being called with a host string instead of full `SSHConfigEntry` object, causing `json: cannot unmarshal string` error
- Fixed in both `handleCreateSession` (clicking server) and `connectSessionIfNeeded` (lazy connect on tab activation)

**Debug Log Tab (permanent feature):**
- Built-in Log tab added to main UI for debugging without browser DevTools
- `dlog()` utility: logs to in-memory buffer (500 max) + console.log
- Copy All button for easy log export
- All drag-and-drop code uses `dlog()` for traceability

**Files Changed:**
- **New**: `frontend/src/utils/dragState.ts`, `frontend/src/utils/debugLog.ts`, `frontend/src/components/log/LogTab.tsx`
- **Modified**: `App.tsx`, `TerminalTab.tsx`, `LocalFileManager.tsx`, `FileManager.tsx`, `AGENTS.md`

**Key Lessons:**
- In Wails with `DisableWebViewDrop: true`, NEVER rely on JavaScript `drop` events ‚Äî they will never fire
- Always use `dragend` + shared memory as the drop mechanism in Wails apps
- `document.elementFromPoint()` is the reliable way to detect drop target in this environment
- An in-app debug log viewer is essential for Wails apps since WKWebView has no DevTools in production
- `ConnectSSH` accepts `SSHConfigEntry` object, not a host string ‚Äî always check Go function signatures

### Project Overview
- **Type**: SSH Terminal File Manager Desktop Application
- **Tech Stack**: Go + Wails v2 (Backend) | React + TypeScript + Ant Design + XTerm.js (Frontend)
- **Platform**: macOS (primary), cross-platform support
- **Current Version**: 2.28

### Key Architecture
- **Backend**: Go with Wails v2 framework, SSH/SFTP via `golang.org/x/crypto/ssh` and `github.com/pkg/sftp`
- **Frontend**: React SPA with TypeScript, Ant Design UI, XTerm.js terminal emulator
- **Communication**: Wails runtime events (`EventsEmit`/`EventsOn`) for real-time terminal I/O
- **Settings**: JSON file in `~/Library/Application Support/xterm-file-manager/settings.json`

### Core Features
1. **SSH Terminal**: Full PTY terminal with real-time I/O (SSH and local sessions)
2. **File Management**: Three-pane layout (Terminal | Remote Files | Local Files) with SFTP transfers
3. **Terminal Features**: Select-to-copy, right-click-paste (MobaXterm-like behavior)
4. **Tools Tab**: JSON Formatter, C Formatter, Escape Tool
5. **Editor Tab**: Multi-tab file editor with drag-and-drop support (Notepad++-like)
6. **Session Management**: Multi-tab interface with connection status indicators

### Key Files
- **Backend**: `main.go` (entry), `internal/app/app.go` (settings/API/file ops), `internal/app/websocket_handler.go` (terminal I/O), `internal/app/ssh_manager.go` (SSH connections), `internal/app/ssh.go` (SSH config parser), `internal/app/local_files.go` (SFTP operations)
- **Frontend**: `App.tsx` (main), `TerminalTab.tsx` (terminal UI), `Terminal.tsx` (xterm component), `EditorTab.tsx` (file editor)
- **Config**: `wails.json`, `go.mod`, `frontend/package.json`

### Recent Changes (v2.22)
- **Editor Tab**: New multi-tab file editor with drag-and-drop support
- Default save directory: `~/Documents/XTermFileManager/`
- Auto-naming: Untitled.txt, Untitled-1.txt, etc.

```go
// internal/app/ssh_manager.go - ListFiles function
// Before: Used ls command via ExecuteCommand
// After: Uses SFTP client directly
func (a *App) ListFiles(sessionID string, path string) ([]FileInfo, error) {
    sftpClient, err := getSFTPClient(sessionID)
    if err != nil {
        return nil, err
    }
    defer sftpClient.Close()
    
    // Resolve ~ to home directory
    if path == "" || path == "~" {
        homeDir, err := sftpClient.Getwd()
        if err != nil {
            return nil, fmt.Errorf("failed to get remote home directory: %v", err)
        }
        path = homeDir
    }
    
    // Read directory via SFTP
    entries, err := sftpClient.ReadDir(path)
    if err != nil {
        return nil, fmt.Errorf("failed to read remote directory: %v", err)
    }
    
    // Convert to FileInfo
    var files []FileInfo
    for _, entry := range entries {
        if entry.Name() == "." || entry.Name() == ".." {
            continue
        }
        files = append(files, FileInfo{
            Name:    entry.Name(),
            Size:    entry.Size(),
            Mode:    entry.Mode().String(),
            ModTime: entry.ModTime().Format(time.RFC3339),
            IsDir:   entry.IsDir(),
        })
    }
    return files, nil
}
```

#### Technical Details of Terminal I/O Fix:

1. **Backend (`websocket_handler.go`)**:

   - Modified `emitTerminalOutput()` to use `runtime.EventsEmit(a.ctx, "terminal:output", payload)`

2. **Frontend (`Terminal.tsx`)**:
   - Import `EventsOn` from Wails runtime
   - Setup event listener: `EventsOn('terminal:output', callback)`
   - When receiving output, write to XTerm: `term.write(payload.data)`
   - Properly cleanup listener on unmount
